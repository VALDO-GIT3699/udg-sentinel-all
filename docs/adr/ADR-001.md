# ADR-001: Selección del Stack Tecnológico Principal

**Estado:** Aceptado  
**Fecha:** 2026-06-29  
**Autores:** Coordinación de Tecnologías de la Información — UDG  
**Contexto:** Plataforma de monitoreo institucional UDG Sentinel  

---

## Contexto

Se necesita construir una plataforma empresarial de monitoreo para más de 200 sitios web de la Universidad de Guadalajara. El sistema debe:

- Escalar de 200 a 2,000+ sitios sin rediseño.
- Operar durante al menos 5 años sin refactorización mayor.
- Ser mantenible por un equipo futuro desconocido.
- Cumplir estándares de seguridad institucionales.
- Producir una interfaz visual al nivel de productos SaaS profesionales.

---

## Decisión 1: Framework backend — **Laravel 12 (PHP 8.2+)**

### Alternativas consideradas

| Alternativa          | Razón de descarte                                                     |
|----------------------|-----------------------------------------------------------------------|
| Drupal como núcleo   | No es un framework de observabilidad. No tiene queue system nativo real. Mezclar CMS con lógica de monitoreo genera deuda técnica insostenible. |
| Symfony puro         | Mayor tiempo de desarrollo. Sin el ecosistema de convenciones y paquetes de Laravel. |
| Node.js / Express    | El equipo actual tiene experiencia en PHP. Mayor riesgo de inconsistencias sin framework opinionado. |
| FastAPI (Python)     | Excelente para microservicios de recolección, pero insuficiente para portal completo con UI, auth, RBAC y reportes. |

### Justificación de Laravel 12

1. **Ecosistema maduro**: Sanctum (auth API), Horizon (queues), Scheduler (cron), Scout (búsqueda), Telescope (debug), Echo (WebSockets) — todo mantenido por el mismo equipo.
2. **nwidart/laravel-modules**: permite arquitectura modular mantenida y documentada.
3. **spatie/***: paquetes de roles/permisos y auditoría con miles de instalaciones productivas.
4. **Eloquent ORM**: expresivo, con soporte de particionado y queries avanzadas sobre PostgreSQL.
5. **PHP 8.2+ con tipos estrictos**: permite código tan seguro y tipado como TypeScript en el backend.
6. **Convención sobre configuración**: el desarrollador que llegue en 4 años sabrá dónde encontrar cada cosa.

### Riesgos y mitigación

| Riesgo                                | Mitigación                                              |
|---------------------------------------|---------------------------------------------------------|
| PHP tiene percepción de "anticuado"   | PHP 8.2 con tipos es moderno. Laravel es top-5 en GitHub stars. |
| Dependencia de un framework           | Arquitectura modular reduce el acoplamiento al framework. |
| Versión mayor en el futuro (Laravel 13+) | Actualizaciones incrementales, sin breaking changes repentinos. |

## Decisión 2: Base de datos — **PostgreSQL 16**

### Alternativas consideradas

| Alternativa    | Razón de descarte                                                              |
|----------------|--------------------------------------------------------------------------------|
| MySQL 8        | Sin soporte nativo real de JSONB, particionado inferior, full-text search limitado. |
| SQLite         | No apto para producción multi-usuario concurrente.                              |
| MongoDB        | Sin JOINs relacionales. Inconsistencia en datos críticos de monitoreo.          |
| TimescaleDB    | Extensión de PostgreSQL válida para series de tiempo, pero agrega complejidad de mantenimiento. Se evalúa para versión 2.0. |

### Justificación de PostgreSQL 16

1. **JSONB nativo**: para metadata de detección tecnológica, breakdown de scores, contexto de alertas — sin columnas dispersas.
2. **Particionado por rango**: crítico para tablas de series de tiempo con millones de filas.
3. **Índices parciales**: reducen el tamaño de índices en tablas grandes filtrando por condición.
4. **ACID completo**: los datos de monitoreo son datos críticos, no pueden perderse ni corromperse.
5. **pg_stat_activity y pg_stat_statements**: monitoreo interno de la base de datos.
6. **Soporte de tipo `inet`**: para almacenar IPs nativamente con validación.
7. **Ampliamente soportado por Laravel/Eloquent**.

## Decisión 3: Cache y colas — **Redis 7**

### Justificación

- **Sesiones**: sesiones distribuidas en Redis permiten escalar la aplicación horizontalmente.
- **Cache**: queries frecuentes del dashboard se cachean con TTL corto (30s–2min) sin cache stale.
- **Queues**: los jobs de recolección se encolan en Redis con Laravel Horizon.
- **Pub/Sub**: base para Laravel Echo + WebSockets para actualizaciones en tiempo real.
- **Locks distribuidos**: evitan que dos workers ejecuten el mismo check simultáneamente.

## Decisión 4: Frontend — **Inertia.js 2 + Vue.js 3 (TypeScript) + Tailwind CSS 4**

### Alternativas consideradas

| Alternativa              | Razón de descarte                                                           |
|--------------------------|-----------------------------------------------------------------------------|
| Blade puro + Alpine.js   | Sin la reactividad necesaria para gráficas en tiempo real y dashboards complejos. |
| Filament v3              | Excelente para admin panels, pero demasiado opinionado para el diseño custom que se requiere. |
| React + API separada     | Duplica la lógica de routing/auth/middleware. Mayor complejidad de mantenimiento. |
| Vue.js + API separada (Nuxt) | Misma razón que React. Inertia mantiene el routing de Laravel sin sacrificar SPA. |
| Bootstrap + jQuery       | No produce la calidad visual de productos SaaS profesionales que se requiere. |

### Justificación de Inertia.js + Vue 3 + Tailwind

1. **Inertia.js**: el routing, auth, middleware y sessions siguen en Laravel. No hay API pública expuesta innecesariamente. Un desarrollador Laravel puede trabajar en el frontend sin aprender un segundo framework de routing.
2. **Vue 3 Composition API + TypeScript**: código frontend tipado, mantenible, con pattern establecido.
3. **Tailwind CSS 4**: utilidades CSS que producen interfaces custom, coherentes, sin bootstrap genérico. Cada componente es original.
4. **ApexCharts**: gráficas profesionales con soporte de series de tiempo en tiempo real, usadas en plataformas como Cloudflare y GitLab.
5. **Heroicons o Lucide**: iconografía consistente y libre.

---

## Decisión 5: Arquitectura modular — **nwidart/laravel-modules**

### Justificación

- Permite que cada dominio (Monitoring, SSL, Security, etc.) tenga su propio ciclo de vida.
- Las migraciones, modelos, controllers y servicios de cada módulo están aislados.
- En el futuro, cada módulo puede convertirse en un microservicio independiente sin reescribir lógica.
- El paquete es ampliamente usado en proyectos Laravel empresariales y tiene mantenimiento activo.

---

## Decisión 6: Containerización — **Docker + Docker Compose**

### Justificación

- **Reproducibilidad**: cualquier desarrollador nuevo tiene el entorno corriendo en < 5 minutos.
- **Paridad staging/producción**: el mismo contenedor que se prueba es el que se despliega.
- **Escalabilidad**: los workers de Horizon escalan independientemente de la aplicación web.
- **Sin dependencia de Laragon en producción**: Laragon es solo para desarrollo local, no para producción.

---

## Consecuencias

### Positivas

- Stack ampliamente documentado en la industria.
- Fácil onboarding de desarrolladores PHP con experiencia en Laravel.
- Ecosistema de paquetes maduro y mantenido.
- Despliegue reproducible y escalable.

### Negativas / Compensaciones

- PHP requiere servidores con PHP-FPM configurado correctamente.
- PostgreSQL requiere más configuración inicial que MySQL.
- Inertia.js es menos conocido que REST + React separados, aunque más simple en la práctica.

---

## Revisión

Este ADR se revisa en cada actualización mayor del stack. El próximo punto de revisión es cuando se alcancen 500+ sitios monitoreados.
